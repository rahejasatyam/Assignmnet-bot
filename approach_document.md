# Conversational SHL Assessment Recommender — Approach Document

## 1. Design Choices and Rationale

The API is built using **FastAPI** to provide a fast, asynchronous, and well-typed interface. The core conversational intelligence is powered by **Groq (llama-3.3-70b-versatile)**, which was chosen for its blazing fast inference speed. Speed is critical as every request must complete within a strict 30-second window, which dictates multiple architectural choices. 

Since the API is entirely **stateless**, the agent parses and resolves the full conversation history natively on the backend during every POST `/chat` request. The backend maintains zero session state or database overhead, making it incredibly resilient and easily horizontally scalable (e.g. on Render/Railway). 

To ensure schema compliance on every request, we leveraged **Pydantic models**. Any LLM response is wrapped in a fallback handler: if the JSON output is missing or malformed, we use RegEx to attempt recovery. If recovery fails, a safe default `ChatResponse` is generated. This guarantees a 0% schema failure rate.

## 2. Retrieval Setup (RAG)

Our data layer is built on the scraped SHL "Individual Test Solutions" catalog (~144 viable technical/personality assessments).

We use **FAISS (IndexFlatIP)** coupled with **sentence-transformers (`all-MiniLM-L6-v2`)**. 
**Why this method?**
- **Speed & Simplicity:** `all-MiniLM-L6-v2` is a 384-dimensional free offline model that is exceptionally fast. FAISS flat indexes compute brute-force exact L2-normalized cosine similarities almost instantly for <1,000 items. There is no need for heavy vector databases like Pinecone or ChromaDB for a dataset of this size. 
- **High Recall Strategy:** We concatenate `Name + Description + Job Levels + Expanded Test Types` into a single dense embedding document per assessment. Realising that hiring managers reveal context sequentially over multiple turns, the Retriever aggregates the texts of **all previous user messages** into one composite query, ensuring that constraints introduced in Turn 2 aren't lost during Turn 5's retrieval.
- **Over-Retrieval:** We query `k=15` from FAISS and pass all 15 results to the LLM. Rather than relying purely on cosine similarity to select the final shortlist, we let the LLM filter the 15 down to 1–10. This significantly boosts **Recall@10**.

## 3. Prompt Design Decisions

The system prompt is the central orchestrator, executing four defined behavioural modes: Clarify, Recommend, Refine, and Compare.

Key mechanisms:
1. **Turn Count Injection:** The prompt explicitly receives the current `{turn_number}`. Rules guide the LLM to strictly clarify during turns 1–2, and forcefully commit to recommendations by turn 6 to avoid hitting the 8-turn hard cap. 
2. **Anti-Hallucination Controls:** We enforce the instruction that URLs must *only* be generated by copying from the retrieved `{catalog_context}`.
3. **Optimizing for Recall:** The LLM is explicitly prompted to "Aim to return ALL relevant assessments from the retrieved list." This prevents the LLM from being over-selective and artificially decreasing Recall@10.

**Code-Level Guardrails:** Even with prompt engineering, LLMs hallucinate URLs. We implemented a post-processing filter that intercepts the LLM's JSON object and strips out any Recommended URL that does not strictly exist in the master `catalog.json` array.

## 4. Evaluation Approach

We implemented an automated test suite using `pytest` and `httpx` to simulate the automated evaluator environment. Evaluation focuses on:

1. **Schema Consistency & Integrity:** We test that `GET /health` and `POST /chat` return exact key-value payloads, and verify that 100% of URLs returned are verifiable SHL catalog URLs.
2. **Behavioural Scenarios:** We crafted multi-turn simulations testing edge cases:
   - *Prompt Injections & Off-Topic:* Asserting `recommendations: []` when asked about recipes or generic HR advice.
   - *Mid-stream Refinement:* Asserting that if a user says "Wait, they actually need a personality test instead", the agent does not restart but seamlessly transitions the shortlist.
   - *User Refusal/Dodging:* Simulating turns where the user responds "I don't know" when asked for seniority levels, forcing the agent to proceed gracefully.

## 5. What Did Not Work (and How It Was Changed)

1. **Wait State Hallucinations for Clarifying Queries:** Initially, when the LLM asked a clarifying question, it would occasionally try to guess URLs or provide placeholder assessments. We fixed this by mandating `recommendations = []` strongly in the system prompt for Clarify stages.
2. **LLM Over-Caution on Turn 1:** Initially, our prompt instructed the agent to *always* clarify on Turns 1 and 2. However, if the user's very first message was highly detailed ("I need a customer service call simulation mapping to competencies for a mid-level hire"), the agent stubbornly refused to recommend, waiting till Turn 3. We altered the system prompt policy to allow immediate Turn-1 committals when context saturation is met.
3. **Messy Scraped Data Noise:** Our initial `requests + BeautifulSoup` scrape pulled in navigation headers and GDPR cookie text within the assessment `description` parameters, corrupting the vector embeddings. We wrote a secondary `fix_descriptions.py` pass to heuristically filter out standard web boilerplate so only relevant assessment features entered FAISS.

## 6. AI Tools Used

- **DeepMind Antigravity Agent:** Used as the primary pair-programming mechanism to rapidly scaffold the FastAPI structure, write the `requests` web scraper, setup the FAISS vector database orchestration, and generate comprehensive unit tests covering all edge case parameters.
- **Groq (Meta Llama-3 70B):** Used to power the dialogue generation for the running API. Chose Llama 70B for its exceptional instruction-following capability in formatting JSON correctly and strictly following our "Refuse" and "Turn Cap" rules.
